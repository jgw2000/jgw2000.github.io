Direct3D 12 - 内存管理

发表于 2022-07-23 更新于 2022-07-24 分类于 Direct3D 12

Memory Management in Direct3D 12

内存管理策略

资源类型

  • Committed Resource

    同时创建虚拟地址空间和物理地址空间 (在托管的物理内存中)，在 Direct3D 12 中应用程序可以将这两部分分开从而更仔细地管理物理内存。

  • Reserved Resource

    在 Direct3D 11 中称为 Tiled Resource，这类型资源有自己唯一的 GPU 虚拟地址空间，这允许应用程序预先分配一个很大的虚拟地址空间，之后再将虚拟地址 pages 映射到 heap 中的特定区域。

    可以通过调用 UpdateTileMappings 将 reserved resource 映射到 heap 中的区域，当虚拟地址范围被映射到 NULL 或者非驻留的 heap，那么这部分的资源也是非驻留的；反之，这部分的资源是驻留的。 Heaps 在创建时候默认是驻留的。

  • Placed Resource

    这是一种更简单的设计，仅仅表示一个指针指向 heap 中的特定区域；通过 Aliasing barriers 可以使用重叠的 placed resources，参考 CreatePlacedResource 和 ResourceBarrier。

内存预算

在 Direct3D 12 中支持更显式地内存分段选择 (在显卡内存和系统内存之间选择)，UMA adapters 只有系统内存。

GPU 不支持页错误，因此开发者必须小心不要过度提交。否则，OS 可能根据物理内存的需求使用更粗粒度的进程调度，例如将 foreground 进程中的某一些页移出，并将需要运行的 background 进程置入分页。

可以通过调用 QueryVideoMemoryInfo 查询内存预算，对于 discrete adapters，“local” 表示显卡内存，“non-local” 表示系统内存；对于 UMA adapters，non-local 总是 0。

Suballocation Within Buffers

上传不同类型资源

在 Direct3D 12 中，可以创建一个 buffer 用于容纳不同类型的资源，然后分别创建单独的 views 将这些不同类型的资源数据绑定到 graphics pipeline 上。

只有当 CPU 只写一次数据并且 GPU 只读一次数据时，才应该使用 upload resources。

在某些情况下

  • GPU 会读多次数据，或者
  • GPU 不会线性的读取数据，或者
  • 渲染已经显著的 GPU-limited

更好的选择可能是使用 ID3D12GraphicsCommandList::CopyTextureRegion 或者 ID3D12GraphicsCommandList::CopyBufferRegion 将 upload buffer 数据复制到一个 default resource

一个 default resource 可以驻留在物理显卡内存。

Buffer alignment

  • 4 MB for multi-sample textures
  • 64 KB for single-sample textures and buffers
