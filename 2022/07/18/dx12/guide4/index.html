Direct3D 12 - 资源绑定 Part 3

发表于 2022-07-18 更新于 2022-07-20 分类于 Direct3D 12

Resource Binding in Direct3D 12

Descriptor Heaps

设计目的

主要目的是提供大部分的内存空间用于储存在一帧或多帧渲染中 shaders 需要引用的所有 descriptors，应用程序可以选择复用 descriptor，或者按顺序分配 heap 空间。

所有 heaps 对 CPU 可见，应用程序也可以请求一个 descriptor heap 需要哪种 CPU 访问属性 - write combined，write back 等

Descriptor heaps 只能被 CPU 即时编辑，不能被 GPU 编辑

同步

Descriptor heap 内容可以在记录命令列表之前、之间以及之后被修改，然而，descriptors 不能被修改当一个已经提交执行的命令列表可能会引用该位置时，因为这可能会导致竞争。

绑定

在任何一个时刻，最多只能绑定一个 CBV/SRV/UAV heap 和一个 Sampler heap，这些 heaps 在 graphics 和 compute pipelines 之间共享。

切换 heaps

应用程序可以在同一个或者不同的命令列表中切换 heaps，使用 SetDescriptorHeaps 和 Reset API。在一些硬件上，这会是一个昂贵的操作，要求 GPU 停下去完成所有依赖当前绑定的 descriptor heap 的工作。因此，应用程序应该试着在 GPU 负载较轻的时候进行此操作，或者在命令列表一开始就限制不再改变 descriptor heap。

Shader 可见 Descriptor Heaps

Shader 可见 Descriptor heaps 可以通过 descriptor tables 被 shaders 引用，主要有 2 类：

  • D3D12_SRV_UAV_CBV_DESCRIPTOR_HEAP
  • D3D12_DESCRIPTOR_HEAP_TYPE_SAMPLER

应用程序可以创建任意数量的 descriptor heaps，shader 可见的 descriptor heap 有硬件大小的限制，而非 shader 可见的 descriptor heap 没有大小限制；如果创建的 shader 可见 descriptor heap 小于硬件大小的限制，驱动程序可能会选择把该 descriptor heap 分配在一个更大的硬件 descriptor heap，这是因为切换硬件上的 descriptor heaps 会导致 GPU 停下等待。

Shader 不可见 Descriptor Heaps

需要 Shader 不可见 descriptor heaps 的原因之一是，这些 heaps 主要是用于暂存以供之后再使用。除此之外，有一些 descriptors 是在记录命令列表的时候直接绑定到 pipeline，只用于传递 view 参数，这些 heaps 永远是 shader 不可见的，包括如下这些：

  • RTVs
  • DSVs
  • IBVs
  • VBVs
  • SOVs

在记录到命令列表之后 (例如调用 OMSetRenderTargets)，存放这些 descriptors 的内存可以立即被复用。

设置 descriptor heaps

在命令列表里可以设置的 descriptor heap 类型有：

  • D3D12_DESCRIPTOR_HEAP_TYPE_CBV_SRV_UAV
  • D3D12_DESCRIPTOR_HEAP_TYPE_SAMPLER

在命令列表里设置的 heap 必须是 shader 可见的

在 bundle 里 descriptor heaps 只能被设置 1 次，并且设置的 heap 必须匹配调用该 bundle 的命令列表的状态，这允许 bundle 继承和编辑命令列表的 descriptor table 设置。

Populating descriptor heaps

当应用程序创建了一个 descriptor heap 之后，其内存中的初始数据是未定义的，可以使用方法直接在 heap 里创建 descriptors 或者从其他地方复制过来。

当应用程序配置一个 descriptor heap 为 CPU 可见，那么 CPU 可以调用方法在 heap 中创建 descriptors 或者复制 descriptor；如果 heap 为 shader 可见，那么从 CPU 读取是不允许的。

Descriptor Tables

每一个 descriptor table 储存 一种或多种类型 descriptors - SRVs，UAVs，CBVs，和 samplers，descriptor table 并没有实际分配内存，只是指向在 descriptor heap 中的偏移和长度。

Descriptor tables 通过命令列表中定义的当前 root signature 绑定到 slot 上

Root Signatures

应用程序通过配置 root signature 将命令列表连接到 shaders 所需要的资源，graphics 命令列表有一个 graphics root signature 和一个 compute root signature，compute 命令列表只有一个 compute root signature，这些 root signatures 之间是互相独立的。

Root parameters 和 arguments

root signature 和一个 API 函数签名类似，决定了 shaders 应该期望的数据类型，但并不实际定义数据或内存。root parameter 是在 root signature 中的一个入口，而在运行时实际设置的 root parameters 的值称为 root arguments；改变 root arguments 将改变 shaders 所读取的数据。

Root constants，descriptors 和 tables

Root signature 包含 3 种类型参数：

  • root constants
  • root descriptors
  • descriptor tables (指向 descriptor heap 中的某一个范围的 descriptors)

Root constants 是一些 32 位的值，在 shader 中显示为 constant buffer

Root descriptors 应该包含最频繁访问的 descriptors，但是仅限于 CBVs，UAV 或者 SRV；Root descriptos 没有大小限制，因此没有越界检查。

Root tables entries 包含 descriptor， HLSL shader 绑定名称和可见性标志。在一些硬件上，设置 descriptors 只对需要的 shader 阶段可见会有性能提升。

Root signature 的布局非常灵活，不考虑硬件水平，应用程序应该使 root signature 尽可能小以最大化效率。

理想情况下，一组 PSOs 会共享同一个 root signature；当一个 root signature 被设置到 pipeline 之后，其所有定义的绑定 (descriptor tables，descriptors，constants) 都可以单独被设置或改变，包括继承到 bundles 中的配置。

使用 Root Signature

应用程序可以通过 API 创建 root signature 对象，在 PSO 中的所有 shaders 必须和 PSO 中指定的 root layout 匹配，或者每一个 shaders 必须包括内置的 root layouts，否则 PSO 创建会失败。

命令列表语义

在命令列表开始执行时，root signature 是未定义的；在 Draw/Dispatch 时，命令列表或 bundle 中设置的 root signature 必须和当前设置的 PSO 匹配；设置 PSO 对象不会改变 root signature，应用程序必须调用专门的 API 设置 root signature

一旦在命令列表中设置了 root signature，就定义了应用程序期望提供的绑定集合，以及在一次 draw/dispatch 调用时使用哪一个 PSO 对象。例如，可以定义一个 root signature 包含如下 entries，每一个 entiry 可以被认为是一个 slot：

  • [0] A CBV descriptor inline
  • [1] A descriptor table containing 2 SRVs，1 CBVs 和 1 UAV
  • [2] A descriptor table containing 1 sampler
  • [3] A 4x32-bit collection of root constants
  • [4] A descriptor table containing an unspecified number of SRVs

Bundle 语义

Bundles 继承命令列表中设置的 root signature 绑定，如果 bundle 需要改变继承的一些绑定，那么必须先设置和命令列表同样的 root signature；如果 bundle 不需要改变任何 root signature 绑定，那么就不需要设置。

bundle 执行完之后，任何绑定的变化都会被继承回命令列表。

Root Signature 限制

内存限制和代价

一个 root signature 的内存限制为 64 DWORDS

  • 每个 descriptor table cost 1 DWORD
  • 每个 root constant cost 1 DWORD
  • 每个 root descriptor cost 2 DWORDs

性能代价

根据间接访问的次数：

  • 每个 root constant cost 0
  • 每个 root descriptor cost 1
  • 每个 descriptor table cost 2

一般来说，遵循如下准则：

  • 使用尽可能小的 root signature，同时平衡使用更大 root signature 的灵活性
  • 在一个更大 root signature 中安排最可能变化的参数，或者需要低访问延迟的参数
  • 如果方便的话，倾向于使用 root constants 或者 root constant buffer views

Static Samplers

Static samplers 的状态是完全定义的并且不可变的，没有内存限制和性能代价。Root signature 可以同时包含 static 和 dynamic samplers，在 descriptor heap 中的 samplers 可以被动态指定和索引。
