Direct3D 12 - 资源绑定 Part 1

发表于 2022-07-17 更新于 2022-07-18 分类于 Direct3D 12

Resource Binding in Direct3D 12

资源绑定概述

资源和 graphics pipeline

Shader 资源（例如纹理，常量表，图像，buffers等）并不是直接绑定到 shader pipeline，而是通过 descriptor 来访问；一个 descriptor 是很小的对象包含了关于一个资源的信息。

Descriptors 组合在一起形成 descriptor tables，每一个 descriptor table 储存一组类型范围内的资源信息。有许多不同类型的资源，最常见的有：

  • Constant buffer views (CBVs)
  • Unordered access views (UAVs)
  • Shader resource views (SRVs)
  • Samplers

其中 SRV，UAV，和 CBV descriptors 可以被组合进同一个 descriptor table。

Descriptor tables 储存在 descriptor heap 中，理想情况下，descriptor heaps 包含所有在接下去几帧内渲染所需要的 descriptors (在 descriptor tables 中)；所有资源储存在用户态的 heaps 中。

另一个概念是 root signature，这是一个应用定义的绑定协定，被 shader 用于定位所需要访问的资源，root signature 可以储存：

  • descriptor heap 中 descriptor tables 的索引
  • 常量
  • 少量 descriptors

换句话说，root signature 提供了性能优化，适用于在每一次绘制中都会改变的少量数据。

资源类型和 views

资源类型

  • Texture1D 和 Texture1DArray
  • Texture2D 和 Texture2DArray，Texture2DMS，Texture2DMSArray
  • Texture3D
  • Buffers (typed，structured and raw)

资源 views

  • Constant buffer view (CBV)
  • Unordered access view (UAV)
  • Shader resource view (SRV)
  • Samplers
  • Render Target View (RTV)
  • Depth Stencil View (DSV)
  • Index Buffer View (IBV)
  • Vertex Buffer View (VBV)
  • Stream Output View (SOV)

其中只有前 4 种 views 对 shaders 可见。

资源绑定的一般流程

  • 创建一个或多个 root signature 对象 - 每一个对应不同的绑定配置需求
  • 创建 shaders 和 pipeline state (使用所需的 root signature 对象)
  • 创建一个或多个 descriptor heaps，包含每一帧渲染所需要的所有 SRV，UAV 和 CBV descriptors
  • 初始化 descriptor heap(s)
  • 对于将要渲染的每一帧
    • 对于每一个命令列表
      • 设置 root signature
      • 更新 root signature 中的常量以及/或者 descriptors (例如世界/视图投影矩阵)
      • 对于每一个将要绘制的 item
        • 定义任何新的每一个对象渲染所需要的 descriptors
        • 更新 root signature 指向 descriptor heaps 中所需要的区域
        • 更新 root signature 中的常量以及/或者 descriptors 用于每一个 item 的渲染
        • 设置 pipeline state (只有在需要改变时)
        • 绘制
      • 重复 (下一个 item)
    • 重复 (下一个命令列表)
  • 重复 (下一帧)

Suballocation

在 Direct3D 12，app 拥有对于内存管理底层的控制；app 可以使用 API 分配一大块内存，之后可以创建 descriptors 指向这一大块内存中的某一个区域，这一过程称为 suballocation。这可以使得 app 更有效率地使用计算和内存，例如，app 可以使用 fences 来决定什么时候一个特定的资源正在被使用或没有被使用。

释放资源

在任何被绑定到 pipeline 上的资源被释放之前，GPU 必须完成对该内存的操作。

最粗糙的方式是等待这一振的渲染完成，从而确定 GPU 已经完成了，更好的方式是使用 fences 来确定 GPU 已经完成。

和 Direct3D 11 绑定模型的区别

内存驻留管理和绑定分离

应用可以显式地控制哪些 surfaces 需要在 GPU 中可以获得并直接使用 (称作 “resident”)，相反地，他们也可以对资源应用其他状态，例如显式地使他们 not resident；重要的一点在于，应用对于资源内存驻留的管理和资源绑定是完全解耦的。

这种解耦降低了系统/硬件在渲染上的成本，因为 OS 不需要频繁地检查局部的绑定状态来确定需要驻留哪些资源。进一步的，shaders 也不再需要知道他们需要引用的具体 surfaces，只要整个可访问的资源集合提前被驻留即可。

对象生命管理和绑定分离

和之前的 API 不同，系统不再跟踪资源的绑定；在释放任何资源之前，例如纹理，应用程序必须确保 GPU 已经完成对该资源的引用，这意味着，GPU 必须已经完成了引用该资源的命令列表的执行。

驱动程序资源状态跟踪和绑定分离

系统不再检查资源绑定来理解什么时候资源状态切换发生了，一个常见的例子是 GPU 和 驱动程序需要知道什么时候一个 surface 从 RTV 切换到 SRV。现在，应用程序必须通过 API 来识别什么时候系统可能关心的任何资源状态的切换正在发生。

CPU GPU 映射的内存同步和绑定分离

系统不再检查资源绑定来理解渲染是否需要被延迟 (因为该渲染依赖于一个资源，其已经被 mapped 用于 CPU 访问但是还没有被 unmapped)；现在，应用程序需要承担起同步 CPU 和 GPU 内存访问的责任，系统提供了机制请求一个 CPU 线程进入睡眠直到工作完成；也可是使用轮询，但是更低效。
