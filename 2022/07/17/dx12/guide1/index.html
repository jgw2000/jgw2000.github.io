Direct3D 12 - 工作提交

发表于 2022-07-17 更新于 2022-09-17 分类于 Direct3D 12

Work submission in Direct3D 12

设计哲学 - 命令队列和命令列表

删除 immediate context

从 Direct3D 11 到 Direct3D 12 最大的改变就是不再有 immediate context，取而代之的是命令列表，我们可以通过命令列表来调用各种渲染 API，绘制图形或者改变渲染状态等。借助于现代多核处理器的优势，我们可以并发的同时记录多个命令列表。

命令列表一般只执行一次，然而，如果应用程序如果能确保前一次执行已经完成的话，那么就可以提交新的执行请求，这样该命令列表就可以多次执行。

组合 GPU 工作项目

借助于现今硬件功能的优势，Direct3D 12 增加了第 2 层命令列表，称为 bundles，其目的在于允许 app 把一小部分 API 命令组合在一起，以便于后续在命令列表中重复执行；在创建 bundle 的时候，驱动程序会尽可能的执行预处理从而使得之后的执行更有效率。

GPU 工作提交

App 现在必须显式地提交命令列表到一个和 Direct3D 设备相关联的命令队列，bundles 不能直接提交到命令队列；任何线程在任何时候都可以提交一个命令列表到任何命令队列，运行时会自动管理保持提交顺序。

创建和记录命令列表和 bundles

创建命令列表

ID3D12CommandAllocator

允许 app 管理分配给命令队列的内存，一个 allocator 可以被用来创建任何数量的 GraphicsCommandList 对象，但是在任一时刻只能关联最多一个当前正在记录的命令队列。

调用 ID3D12CommandAllocator::Reset 回收分配的内存，但在回收之前，app 必须确保 GPU 不再执行任何相关联的命令列表。

ID3D12PipelineState

为命令队列指定初始的 pipeline state，app 通常在初始化时候会创建很多 ID3D12PipelineState 对象，之后通过调用 ID3D12GraphicsCommandList::SetPipelineState 切换当前绑定的状态对象。

注意：bundles 不会继承其所在的命令列表中设置的 pipeline state ！

记录命令列表

命令列表创建或者调用 ID3D12GraphicsCommandList::Reset 之后即进入记录状态，命令列表在执行过程中也可以调用 Reset，一个典型用法是在提交之后立马 Reset 来复用分配给其他命令队列的内存。

调用 Close 退出记录状态。

一个简单的 allocator 池化策略大约需要 numCommandLists * MaxFrameLatency 个 allocators；一个更高级的池化策略是在同一个线程里复用 allocators，大约需要 numRecordingThreads * MaxFrameLatency 个 allocators。

使用 fence 来决定什么时候一个给定的 allocator 可以被复用。

由于命令列表在提交之后可以立马 Reset，因此在调用 ID3D12CommandQueue::ExecuteCommandLists 之后可以直接放回 pool 中。

执行和同步命令列表

命令队列

Direct3D 12 命令队列代替了 immediate 模式的运行时和驱动程序同步，使用 API 显式地管理并发、并行和同步，给开发者提供了以下一些改进：

  • 允许开发者避免由于不必要的同步导致的低效
  • 允许开发者在更高的层次、更有效率、更准确地使用同步
  • 显式地标明昂贵的操作

从多个命令队列访问资源

运行时附加以下这些规则来限制资源的访问：

  1. 一个资源不能同时被多个命令队列写入，当一个资源在一个队列中切换到可写状态之后，可以认为被该队列独占拥有，只有在切换到 READ 或者 COMMON 状态之后才能被其他队列访问
  2. 当一个资源在 READ 状态，可以同时被多个队列读取

使用 Fences 同步命令列表执行

当一个队列中的命令列表依赖另一个队列正在操作的资源时，app 需要显式地管理这两个队列之间的同步。Direct3D12 使用 Fences 概念来完成队列之间的同步，在 API 中由 ID3D12Fence 接口表示。

管理 Graphics Pipeline State

概述

当几何被提交到 GPU 绘制时，有一些硬件设置决定如何解释和渲染这些输入数据，这些设置称为 graphics pipeline state，常用的设置包括：

  • 光栅化状态
  • 混合状态
  • 深度模板状态
  • 几何拓扑类型
  • shaders

在 Direct3D 12 中大多数 graphics pipeline state 通过 pipeline state objects (PSO) 来设置，app 一般在初始化的时候创建任意多的这些对象 (由 ID3D12PipelineState 接口表示)。之后在渲染的时候，命令队列可以通过调用 ID3D12GraphicsCommandList::SetPipelineState 快速地切换不同的 pipeline state。

在现今的 GPU 硬件中，不同的单元之间会有依赖关系，例如，混合状态单元会依赖光栅化状态单元；因此，Direct3D 12 在设计上允许 GPU 在每一个 pipeline state 中预处理所有这些有依赖关系的设置，通常在初始化的时候，从而使得在渲染时尽可能有效率地在不同状态之间进行切换。

Graphics Pipeline State 继承

对于使用 PSO 设置的 Graphics Pipeline States ，其中的任何状态都不会被命令列表或者 bundles 继承。命令列表和 bundles 在创建的时候通过调用 ID3D12Device::CreateCommandList 来设置初始状态，如果没有指定 PSO，那么就会使用默认的初始状态；可以通过调用 ID3D12GraphicsCommandList::SetPipelineState 改变当前 PSO。

命令列表也不会继承通过 ID3D12GraphicsCommandList 接口方法设置的状态，例如 RSSetViewports。

Bundles 继承所有不使用 PSO 设置的状态，除了几何拓扑类型，当一个 bundle 开始执行时，集合拓扑类型总是被设置为 D3D12_PRIMITIVE_TOPOLOGY_TYPE_UNDEFINED。在 bundle 里设置的任何状态 (PSO，非 PSO 状态以及资源绑定)都会影响其所在的命令列表的状态。

在命令列表或者 bundles 中设置的资源绑定状态会继承。
